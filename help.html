Mission Control Platform

Central operations platform for device audit, compliance management, network monitoring, and security operations. Accessible at audit.simplyict.com.au.

The platform serves early childhood education services across multiple sites, providing tools for ACECQA compliance auditing, network probe monitoring, and SOC threat detection.

Login & 2FA

All pages require authentication. The login system includes brute-force protection and optional two-factor authentication.

  1. Navigate to the login page and enter your username and password.
  2. If 2FA is enabled, you will be redirected to the verification page. Open your authenticator app and enter the 6-digit code.
  3. If 2FA is not yet set up, go to 2FA Setup to scan the QR code with Google Authenticator or Authy.
After 5 failed login attempts, your account is locked for 15 minutes. Contact an admin if you are locked out.

Admins can reset 2FA for any user from the User Management page.

Navigating the Platform

The platform is organised into sections accessible from the top navigation bar or the Platform Home page:

  • Audit Centre — Device audits, reports, active device register, cleanup tools
  • Executive — Director dashboard, governance summary, audit recency
  • Compliance — Site onboarding, incident register, remediation tracking
  • Network Monitoring — Live dashboards, all-sites view, device classification, alerts
  • Security Operations — Wazuh SOC, SOC agent dashboard, Telegram integration
  • Administration — User management, 2FA setup, user guide
  • Projects — Project Hub, project manager, project trackers

Each page has a Home button to return to the platform home, and Help and DevDocs buttons for assistance.

Audit Running a Device Audit

The Device Audit form lets you record device checks at a site and classify each device against compliance criteria.

  1. Go to Device Audit Form and select a site from the dropdown.
  2. If a previous audit batch exists for this site, you can continue it. Otherwise, start a new batch.
  3. For each device, record: device type, serial number, room, OS version, photo count and date, OneDrive sync status, security check, and notes.
  4. Click Save All Rows to submit the entire batch at once. Individual rows can also be saved.
  5. The system classifies each device as High Risk, Needs Attention, or On Track based on thresholds configured per site.
The Data Completeness column shows whether each device has all required fields filled (green check = complete, yellow warning = partial, red cross = empty).

Reports Generating PDF Reports

Executive summary PDF reports can be generated for any completed audit batch.

  1. Go to Audit Reports and select a site and audit batch.
  2. Click Generate PDF to produce an executive report with cover page, KPI summary, risk breakdown, room-by-room results, and a Policy Compliance Checklist.
  3. The PDF downloads automatically. Filename format: {site}-{date}-executive.pdf.

The report includes: site thresholds, audit frequency, photo retention days, policy compliance score, and NQS quality area mappings.

Devices Active Device Register

The Active Device Register shows all devices currently registered in the system with their latest audit status. Use this page to get a quick overview of device coverage across all sites.

Records Device Records & CSV

The Device Records page provides full device inventory management:

  • Export CSV — Download all device records as a 26-column CSV file
  • Import CSV — Upload a CSV file to bulk-upsert devices (matched on device_id)
  • Create Device — Add a single device manually
  • Edit — Update device details inline
CSV import uses upsert logic — existing devices are updated, new devices are created. Download the current CSV first to see the expected column format.

Compliance Site Onboarding

Site Onboarding configures ACECQA policy compliance settings per site. Each site has 9 policy toggles:

  • Consent forms current
  • Photo policy documented
  • Staff privacy training current
  • Device encryption enforced
  • Access controls documented
  • Data retention policy in place
  • Incident response plan exists
  • Privacy impact assessment completed
  • Third-party data agreements in place

Additional fields: audit frequency (days) and photo retention days/thresholds. Save to persist changes.

Compliance Incident Register

The Incident Register tracks incidents with full CRUD operations:

  • Create — Record new incidents with severity levels and mandatory-reportable flags
  • Edit — Update incident details, status, and notes
  • Delete — Remove incidents with confirmation prompt

Compliance Remediation Tracker

The Remediation Tracker manages remediation actions with:

  • Priority and severity tracking (Critical, High, Medium, Low)
  • Overdue detection with colour-coded status indicators
  • State management: Open, Resolved, Accepted Risk
  • Target date assignment and progress tracking

Dashboards Director Dashboard

The Director Dashboard provides executive oversight with 11 KPI tiles and per-site breakdowns:

  • Remediation KPIs — Open, overdue, resolved, critical, high priority counts
  • Incident KPIs — Open, mandatory, critical, high severity counts
  • Policy KPIs — Policy gaps count, sites with complete policy coverage
  • Per-Site Policy Table — Confirmed/9 policies, gaps, missing policy names per site
  • Audit Recency Table — Last audit date and age per site, overdue flag >90 days
  • NQS Mapping — Findings mapped to NQS 2.2, 7.1, 7.2 quality areas

Click Refresh All to reload both director and governance data.

Dashboards Governance Summary

The governance summary endpoint (GET /governance-summary) powers the director dashboard. It returns aggregated data on remediation, incidents, policy compliance, and audit recency.

Monitoring Live Dashboard

The Live Dashboard provides a per-site view of network devices, alerts, agents, and observations:

  • Site Selector — Choose a site from the dropdown to view its data
  • KPI Cards — Total devices, online, offline, unknown, open alerts, agent status
  • Device Table — IP, MAC, friendly name/hostname, device type, latency, last seen
  • Alerts List — Open and acknowledged alerts with severity, title, source
  • Agent Status — Online/offline probe agents for the selected site

Requires the API base URL and API key to connect to the backend monitoring API.

The API key and base URL are saved in your browser's local storage. You only need to enter them once.

Monitoring All Sites Overview

The All Sites Dashboard shows a global view of all monitored sites with health status, device counts, online/offline breakdown, alert counts, and agent status. Use this page to quickly identify sites that need attention.

Monitoring Device Classification

The Device Admin page lets you classify devices discovered by the probe agent:

  • Friendly Name — Set a human-readable name for the device
  • Device Type — Choose from: Unknown, Network Infrastructure, Windows/Linux/Apple Device, Mobile, Printer, VoIP Phone, Camera, Server, IoT Device
  • Criticality — Set to LOW, MEDIUM, HIGH, or CRITICAL
  • Expected Online — Check if this device should always be online (enables offline alerting)
  • Notes — Add free-text notes
  • Guess — Auto-classify using AI based on hostname, MAC OUI, and vendor
Classifying devices with meaningful names and criticality improves the accuracy of offline alerts and the executive dashboard.

Monitoring Alert Management

The Alerts page provides full alert lifecycle management:

  • Acknowledge — Mark the alert as seen (moves from OPEN to ACKNOWLEDGED)
  • Resolve — Close the alert when the issue is fixed
  • Suppress — Silence future occurrences of this alert type
  • Reopen — Re-open a resolved or suppressed alert

Alert types include: DEVICE_OFFLINE, DEVICE_BACK_ONLINE, AGENT_MISSING. Each alert has severity (CRITICAL, HIGH, MEDIUM, LOW), a fingerprint for deduplication, and a count of occurrences.

Monitoring Executive Dashboard

The Executive Dashboard provides multi-site intelligence with availability trends, infrastructure summaries, and device reduction targets. Shows critical devices offline, site health rankings, and executive-level KPIs.

Monitoring Probe Agent Setup

Probe agents run on Intel NUCs at client sites. They perform periodic ARP + ICMP discovery of the local subnet and report device state to the central backend.

Windows Install

Run as Administrator in PowerShell:

.\install.ps1 -ApiKey <key> -ApiBase https://audit.simplyict.com.au/monitoring-api -SiteId <id> -SiteName "Name" -AgentId <agent> -Subnet "192.168.1.0/24"

Linux Install

Run as root:

sudo ./install.sh --api-key <key> --api-base https://audit.simplyict.com.au/monitoring-api --site-id <id> --site-name "Name" --agent-id <agent> --subnet 192.168.1.0/24

Diagnostics

On Windows, run .\probe-diagnose.ps1 to check installation paths, Python environment, service status, log files, and API reachability.

The installer uses embeddable Python (no system Python dependency). It downloads NSSM, Python, and the agent package automatically.

SOC Wazuh SOC Dashboard

The Wazuh SOC Dashboard is a full SIEM single-page application with 14 views proxied through the platform:

  • Command Center — KPI tiles (agents, vulnerabilities, SCA score, threat index) with Chart.js charts
  • Agents — Filterable agent list with drill-down to SCA, FIM, vulnerabilities, inventory tabs
  • SCA Compliance — Fleet-wide Security Configuration Assessment scores and policy results
  • File Integrity — FIM event log across all agents
  • Vulnerabilities — CVE aggregation with severity KPIs and sortable table
  • MITRE ATT&CK — Technique heatmap grid
  • Rules & Decoders — Wazuh rule/decoder catalogue browser
  • Events & Alerts — Real-time alert feed with severity filtering
  • Topology — OS distribution and agent version charts
  • Threat Intel — AlienVault OTX integration status and IOC counts
  • SOC Autopilot — AI-powered case management
  • Manager Health — Daemon status, manager info, cluster status
  • Groups — Agent group membership

SOC SOC Autopilot Agent

The SOC Agent Dashboard provides an AI-like auto-triage agent that monitors alerts, auto-resolves false positives, triggers autopilot cases, and sends periodic summaries.

  • Monitors open alerts every 5 minutes
  • Auto-resolves DEVICE_OFFLINE alerts when devices come back online
  • Escalates persistent critical alerts with detailed incident information

SOC Telegram Integration

The platform integrates with Telegram for real-time SOC notifications:

  • Receive real-time alerts directly in a Telegram chat
  • Use commands: /status, /alerts, /cases, /help
  • Daily SOC digest at 08:00 with overnight summary and trends
  • 30-minute escalation for persistent CRITICAL alerts

Projects Managing Trackers

The Project Hub is the master tracker panel for all build project trackers. It provides an overview of all projects with health status, task progress, and deployment controls.

Each project card shows:

  • Project name, description, and health status (online/offline/unknown)
  • Task progress bar with done/total counts
  • Folder path, server host, and SSH user badges
  • Open Tracker button (if a tracker is deployed)
  • Deploy Tracker button (if no tracker exists)

Projects Project Tasks & Kanban

Click any project card to view its tasks in list or Kanban view:

  • List View — Filterable task table with status, priority, owner, and target date
  • Kanban View — Drag-and-drop board with columns: To Do, In Progress, Blocked, Review, Done
  • Add Task — Quick-add tasks with title, priority, owner, and target date
  • CSV Export — Download tasks as a CSV file

Projects Deploying a Tracker

For projects without a tracker, click Deploy Tracker to scaffold and deploy a standalone tracker server:

  • Local — Create tracker files on the local server, install systemd service, start it
  • Remote — SCP tracker files to a remote server via SSH, install and start the service

The deploy modal shows real-time progress logs. After deployment, the health check verifies the tracker is responding.

Tracker deployment creates 4 files: index.html, server.py (stdlib HTTP server), data.json, and a .service unit file.

Troubleshooting

Blank page after login

Hard refresh (Ctrl+Shift+R on Windows/Linux, Cmd+Shift+R on Mac) or clear your browser cache. Also check your session hasn't expired.

Wazuh SOC pages not loading

Check that the Wazuh API proxy is running: sudo systemctl status mission-control-ui. The proxy at /wazuh-api/{path} forwards requests to the Wazuh SOC API on localhost:8000 and auto-retries on 401.

Devices showing offline

Check the probe agent status on the Live Dashboard. If the agent is offline, the site NUC may have network issues or the agent process may have stopped. On the NUC, run: nssm status MissionProbe.

CSV export not working

Ensure your browser allows pop-ups for file downloads. The export creates a Blob download — if pop-ups are blocked, the download will be silently prevented.

2FA code not working

Ensure your device's time is correct (auto-sync recommended). TOTP codes are time-based — a clock drift of more than 30 seconds will cause codes to be rejected. Contact an admin to reset your 2FA if needed.

PDF report shows "N/A" for data

This usually means the audit batch doesn't have complete device entries, or the site doesn't have threshold settings configured. Check the Site Onboarding page for threshold values.

Login returns "Invalid credentials"

Check Caps Lock is off. If you've forgotten your password, contact an admin to reset it. Remember the 5-attempt lockout — wait 15 minutes if locked out.

If you encounter persistent issues, check the system status at the Project Hub or contact SimplyICT support.

Service Statuses Explained

Device Statuses

StatusMeaning
ONLINEDevice responded to ping or ARP in the latest scan cycle
OFFLINEDevice expected online but not seen for >180s and ping verification failed
UNKNOWNNewly discovered device, not yet classified
DEGRADEDDevice online but with issues (high latency, packet loss)
DISABLEDDevice excluded from monitoring (expected_online = false)

Alert Severity Levels

SeverityMeaning
CRITICALImmediate attention required — security breach, extended outage
HIGHUrgent — device offline with stable history, pattern escalation
MEDIUMNeeds review — routine offline detection, non-critical
LOWInformational — device back online, agent reconnected

Remediation Priorities

PriorityMeaning
CriticalMust be resolved immediately — compliance violation, data breach
HighShould be resolved within 7 days — significant risk
MediumShould be resolved within 30 days — moderate risk
LowResolve when convenient — minor risk

Agent Statuses

StatusMeaning
ONLINEAgent sent heartbeat within the expected interval
OFFLINENo heartbeat received for >300s (configurable)
