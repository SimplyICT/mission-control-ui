A: Extend project-manager.html

Add project site links + "New Tracker" wizard to the existing page. Projects in the grid get a "Open Tracker" button. New project wizard adds server/folder fields. Keeps the kanban/task views underneath.

Pros

  • Single URL to remember
  • Leverages existing API/code
  • Users already know it

Cons

  • Page has dual purpose (hub + tasks)
  • Needs backend schema changes

B: New project-hub.html page

Clean dedicated page showing all project sites with their tracker links, server info, and status. Separate "New Project Site" creation flow. Links to project-manager for task drill-down.

Pros

  • Clean separation
  • Designed purely for this

Cons

  • Another URL to learn
  • Duplicate project data

C: Rebuild index-platform.html section

Add a "Build Projects" section to the existing landing page with tracker links and creation wizard. Keeps everything under one roof.

Pros

  • Central hub already

Cons

  • Page already very full
  • Not the right place for creation flows

My Recommendation: Option A

Extend project-manager.html — it already has the project concept, CRUD API, and users know it. We add:

  1. Tracker URL field to each project (stored in projects.json)
  2. "Open Tracker" button on each project card — links to the tracker URL
  3. "New Project Site" wizard with fields for: server IP, port, folder path, tracker URL
  4. Create Tracker Scaffold — option to auto-generate a tracker index.html + server.py in the target folder
  5. Status indicator — ping the tracker URL to show online/offline
